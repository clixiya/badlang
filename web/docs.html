Introduction #

bad is a CLI-first API testing domain-specific language (DSL) written in C. It lets you write expressive, readable test suites for HTTP APIs without a heavyweight framework, runtime, or package ecosystem.

Tests in bad feel like documentation — you describe what a request should look like and what response to expect, in plain declarative statements. The bad binary runs these .bad files and reports results with colored terminal output, timing info, and structured history.

Philosophy: bad is designed for engineers who want fast, readable API tests that live close to their code — not a separate test platform to manage.

Key Capabilities

  • Full HTTP: GET, POST, PUT, PATCH, DELETE with body and header blocks
  • JSON path assertions with comparison operators and string matchers
  • Variable capture from response bodies, environment, CLI args, and object literals
  • Request/template reuse with inline expectations and override controls
  • If/else control flow, retry policies, named latency timers
  • Lifecycle hooks plus error hooks and URL wildcard hooks
  • Grouped tests, skip/only filters, fail-fast mode
  • Multiple output formats: default, verbose, flat, table, JSON
  • JSONL history logging with optional step-level detail
  • VS Code extension with syntax highlighting and snippets

Installation #

bad is distributed as a compiled C binary. Clone or download the source, then build with make. No runtime or package manager needed.

macOS

brew install curl
make
./bad examples/01-basics/quick_start_demo.bad

Linux (Debian/Ubuntu)

sudo apt update
sudo apt install -y build-essential libcurl4-openssl-dev
make
./bad examples/01-basics/quick_start_demo.bad

Windows (MSYS2/MinGW)

pacman -S --needed mingw-w64-ucrt-x86_64-gcc mingw-w64-ucrt-x86_64-curl make
make
./bad.exe examples/01-basics/quick_start_demo.bad

PowerShell runner (alternative):

.\run_bad.ps1 examples/01-basics/quick_start_demo.bad

Global Install

To install bad system-wide and use it from any directory:

sudo make install
bad examples/01-basics/quick_start_demo.bad

VS Code Extension

Install the bad-language-tools VSIX from the extenstion/ folder in the repo. It provides syntax highlighting, snippets, and a custom file icon theme for .bad files.

code --install-extension extenstion/dist/bad-language-tools-1.1.0.vsix --profile comp
Note: bad requires libcurl for HTTP. On macOS, Homebrew's curl works. On Linux, install libcurl4-openssl-dev before building.

Quick Start #

Create a file called quick.bad with your first test:

test "ping" {
    send GET "https://jsonplaceholder.typicode.com/users/1"
    expect status 200
}

Run it:

./bad quick.bad

You'll see colored output showing if the test passed or failed.

Using base_url

Set a base URL at the top of your file to avoid repeating the full server address:

base_url = "https://jsonplaceholder.typicode.com"
timeout  = 10000

test "get user" {
    send GET "/users/1"
    expect status 200
    expect json.id == 1
    expect json.name exists
}

test "list posts" {
    send GET "/posts"
    expect status 200
    expect json.0.id exists
}
Tip: Run your first example with ./bad examples/01-basics/quick_start_demo.bad to see bad in action against a live public API.

Test Blocks #

The test block is the fundamental unit in bad. Each test has a name and a body containing one or more send statements and expect assertions.

test "name of the test" {
    send GET "/endpoint"
    expect status 200
    expect json.field exists
}

Tests run sequentially in the order they appear in the file. Variables set inside a test are accessible in subsequent tests (file-scoped).

Skip a Test

skip test "legacy endpoint" because "deprecated"

Focus a Test

only test "smoke test"

When any only test exists, only those tests run.

Requests — send #

The send keyword fires an HTTP request. Supported methods: GET, POST, PUT, PATCH, DELETE.

Basic Usage

# Simple GET
send GET "/users/1"

# GET using a variable as path
let path = "/users/1"
send GET path

# POST with full URL
send POST "https://api.example.com/posts"

Request Body

send POST "/posts" {
    body {
        title:  "Hello bad"
        body:   "testing is easy"
        userId: 1
    }
}

# Alias: payload = body
send POST "/posts" {
    payload {
        title: "Hello"
    }
}

Request Headers

send GET "/private" {
    header {
        Authorization: "Bearer my-token"
        Accept:        "application/json"
        X-Custom:      "bad-lang"
    }
}

# Alias: headers = header
send GET "/endpoint" {
    headers {
        Content-Type: "application/json"
    }
}

Combined Body and Headers

send POST "/users" {
    body {
        name:  "Krish"
        email: "krish@example.com"
    }
    header {
        Authorization:  auth_token
        Content-Type:   "application/json"
        X-Request-Id:   "req-001"
    }
}
Path variables: You can use a variable name directly as the path: send GET user_path. bad resolves it at runtime.

Object Sources in Body/Header

let common_headers = {
    Accept: "application/json"
    X-Client: "bad"
}

let payload_defaults = {
    userId: 1
    completed: false
}

send POST "/todos" {
    header common_headers
    body {
        payload_defaults
        title: "override title"
    }
}
Object variables: Object variables are spread-only in body/header sources. They cannot be resolved as scalar values.

Assertions #

Assertions verify the response. Use expect or assert (they're identical). A failed assertion marks the test as failed but continues running unless stopped.

Status Assertions

expect status 200
expect status 201
assert status 204
    expect status >= 200
    expect status < 500
expect status in [200, 201, 204]

JSON Path Assertions

# Existence
expect json.id exists
expect json.user.name exists
expect json.items.0.price exists

# Equality
expect json.id == 1
expect json.status == "active"
expect json.verified == true

# Comparison
expect json.count > 0
expect json.count >= 1
expect json.price < 100
expect json.score <= 99.9

# Inequality
expect json.status != "error"
expect json.id != 0

String Matchers

expect json.email contains "@"
expect json.name starts_with "Krish"
expect json.email ends_with "@example.com"
expect json.trace_id regex "^[a-z0-9-]{8,}$"

Timing Assertions

# Assert last request was under 500ms
expect time_ms < 500

# Assert named timer
expect time auth_flow < 2000
JSON paths use dot notation. Array items use zero-based indices: json.items.0.id accesses the first item's id.

Variables — let #

Variables are declared with let and are file-scoped. They persist across test blocks, enabling multi-step workflows like authentication.

Basic Declarations

let user_id    = "1"
let base_path  = "/users"
let timeout_ms = 5000
let enabled    = true

Capture from Response

After a send, these sources are available to let:

send POST "/auth/login" {
    body { email: "user@example.com" password: "secret" }
}

let token   = json.token          # JSON field from response body
let code    = status               # HTTP status code (number)
let elapsed = time_ms              # Duration of last request in ms
let now     = now_ms               # Current epoch timestamp in ms
let auth    = bearer token         # Prepends "Bearer " if missing

Environment and CLI Args

let api_key  = env API_KEY         # Read process env variable
let mode     = env DEPLOY_ENV
let first    = args 0              # First positional CLI argument
let second   = args 1

Object Variables

let shared_headers = {
        Accept: "application/json"
        X-Trace: "suite-1"
    }

    let base_body = {
        userId: 1
        completed: false
    }

    send POST "/todos" {
        header shared_headers
        body base_body
    }

Printing Variables

print token
    print elapsed
    let user_name = json.name
    print user_name
Scope: Variables are file-scoped. They persist from the moment they are set until the file finishes. A variable set inside a test is visible in all subsequent tests.

Top-level Options #

Set runtime options at the top of any .bad file. CLI flags override these values.

base_url       = "https://api.example.com"
timeout        = 10000       # Request timeout in milliseconds
print_request  = false       # Log outgoing request details
print_response = true        # Log raw response body
show_time      = true        # Show per-test timing
show_timestamp = true        # Show wall-clock timestamps
json_pretty    = true        # Pretty-print JSON in output
save_history   = true        # Write results to JSONL history
save_steps     = true        # Include step detail in history
history_mode   = "per-file" # all | per-file | per-test | off
history_methods = "GET,POST" # save only selected methods
history_exclude_methods = "DELETE"
history_only_failed = false
history_include_headers = true
history_include_request_body = true
history_include_response_body = true
history_max_body_bytes = 0   # 0 => unlimited
history_dir    = ".bad-history"
history_file   = ".bad-history/all-runs.jsonl"
retry_count    = 2           # Global default retry count
retry_delay_ms = 100         # Global retry delay
retry_backoff  = linear      # linear or exponential
retry_jitter_ms = 25         # Jitter range in ms
Aliases: base is an alias for base_url, and wait is an alias for timeout.

Conditionals — if / else_if / else #

bad supports if/else control flow both inside test blocks and at the top level of a file.

test "conditional logic" {
    send GET "/health"

    if status == 200 {
        print "health check passed"
    } else_if status == 429 {
        sleep 100
        stop because "rate limited"
    } else {
        stop_all because "service unhealthy"
    }
}

# Logical operators
if status == 200 and not json.error exists {
    print "healthy response"
}

if status == 200 or status == 201 {
    print "accepted status"
}

# Explicit grouping with parentheses
if (status == 200 or status == 201) and not (json.error exists) {
    print "success without errors"
}

Top-level Conditionals

Useful for suite preflight checks that abort early:

let preflight = send GET "/health"

if status != 200 {
    stop_all because "health check failed"
} else {
    print "preflight ok"
}

Conditional Skip / Fail

# Skip current test if condition is met
skip_if status != 200 because "service unavailable"

# Mark current test as failed if condition is met
fail_if status >= 500 because "server error"

Retry Policy #

bad automatically retries requests on network failures, HTTP 429, and 5xx responses. Configure retry behavior globally or per-request.

Global Defaults

retry_count    = 2
retry_delay_ms = 100
retry_backoff  = linear     # linear | exponential
retry_jitter_ms = 25

Per-request Override

send GET "http://unreliable.example.com/data" {
    retry 3
    retry_delay_ms 200
    retry_backoff exponential
    retry_jitter_ms 50
}
OptionDescriptionDefault
retry NNumber of retry attempts0
retry_delay_ms NDelay between retries in ms0
retry_backofflinear or exponentiallinear
retry_jitter_ms NRandom jitter added to delay0
Exponential backoff: Each retry doubles the previous delay. Combined with jitter, this avoids thundering-herd problems in distributed tests.

Timing & Latency #

bad has built-in latency tracking both for individual requests and multi-step flows.

Built-in Values

ValueDescription
time_msDuration of the most recent request in milliseconds
now_msCurrent epoch time in milliseconds
time <n>Elapsed ms for named timer <n>
<n>_msAuto-variable written when time_stop <n> runs
last_time_msAlias for time_ms of most recent request

Named Timers

time_start checkout_flow

# ... multiple requests ...
send POST "/cart/add" { body { sku: "abc" qty: 1 } }
send POST "/checkout"
send GET "/order/confirm"

time_stop checkout_flow

let total = time checkout_flow
print total

expect time checkout_flow < 5000

Latency Assertions

expect time_ms < 300
expect time auth_flow < 2000
expect time_ms <= 1000

Stop & Sleep #

Control test execution flow with stop commands and pacing helpers.

# Stop current test, continue with next
stop because "precondition failed"

# Abort entire file execution
stop_all because "critical failure"

# Pause execution for N milliseconds
sleep 250
KeywordEffect
stopExits the current test block, moves to next test
stop_allExits the entire file, no more tests run
sleep NPauses execution for N milliseconds
skip_if cond because "reason"Conditionally skips test if condition holds
fail_if cond because "reason"Marks test as failed if condition holds

Groups #

Groups organize related tests under a named section. They appear as a labeled block in output.

group "users" {
    test "get one user" {
        send GET "/users/1"
        expect status 200
    }

    test "list users" {
        send GET "/users"
        expect status 200
    }
}

group "posts" {
    test "get post" {
        send GET "/posts/1"
        expect status 200
    }
}

# Skip an entire group
skip group "legacy" because "deprecated" {
    test "old endpoint" {
        send GET "/v1/users"
        expect status 200
    }
}

# Run only one group
only group "users"
Nesting: Groups cannot be nested inside other groups. Place them at the top level of the file.

Lifecycle Hooks #

Hooks run at predictable points in execution. They can contain any valid bad statements.

before_all {
    print "suite setup"
    let trace_id = "suite-001"
}

before_each {
    let start = now_ms
    print "test starting"
}

after_each {
    let end = now_ms
    print "test done"
}

after_all {
    print "suite complete"
}

test "first test" { ... }
test "second test" { ... }

Error Hooks

on_error {
    print "any assertion/network failure"
}

on_assertion_error {
    print "assertion failure"
}

on_network_error {
    print "network failure"
}

URL Wildcard Hooks

before_url "/*" {
    print "before request"
}

after_url "/users/*" {
    print "after users request"
}

on_url_error "/users/*" {
    print "users request failed"
}

URL hook patterns support * wildcards and are matched against both full URL and request path.

HookWhen it runs
before_allOnce before all tests in the file
before_eachBefore every individual test
after_eachAfter every individual test
after_allOnce after all tests in the file
on_errorWhen assertion or network failures happen
on_assertion_errorOnly on assertion failures
on_network_errorOnly on network/transport failures
before_url "pattern"Before requests matching URL/path wildcard
after_url "pattern"After requests matching URL/path wildcard
on_url_error "pattern"On errors for matching URL/path wildcard

Imports & Exports #

bad supports modular test suites through import/export declarations. Share variables, request templates, and config across files.

Exporting

# shared.bad
export let base = "https://api.example.com"
export let timeout = 10000
export let auth_token = "Bearer xyz"

export request get_user {
    method GET
    path "/users/1"
    header {
        Accept: "application/json"
    }
}

Importing

# Import everything
import "shared.bad"

# Alias: use = import
use "shared.bad"

# Selective import
import "shared.bad" only auth_token, base

# Selective import with alias
import "v2/templates.bad" only users_path as user_path

# Import only to execute (run its tests)
only import "smoke.bad"
Paths: Import paths are relative to the current working directory when bad is invoked, not relative to the importing file.

Request Templates #

Request templates define reusable request configurations. Use them to avoid duplicating headers, paths, expectations, and retry policy across tests.

Define a Template

template get_user {
    method GET
    path "/users/1"
    header shared_headers
    expect status 200
  }

  template create_todo {
    method POST
    body defaults
    retry 2
    retry_delay_ms 100
    expect status in [200, 201]
}

  # Aliases: req = request, template = request
req create_post {
    method POST
    path "/posts"
    header {
        Content-Type: "application/json"
    }
}
Template path: path is optional in templates. You can supply it later at send time via with { path ... }.

Using a Template

test "fetch user" {
    send req get_user
    expect status 200
}

# Override fields at call site
test "fetch user 2" {
    send req get_user with {
        path "/users/2"
        headers {
            X-Version: "2"
        }
    }
    expect status 200
}

# Merge template body defaults with override body fields
send req create_todo with {
  body {
    title: "override"
  }
  body_merge true
}

# Also valid
send request get_user
send template get_user

By default, a body override replaces template body content. Use body_merge true (or merge_body true) to merge template defaults with override fields.

Template Filters

# Run only tests that use these templates
only req get_user, create_post

Only & Skip Filters #

bad provides fine-grained control over which tests, groups, imports, and templates execute.

# Run only a named test
only test "smoke test"

# Run only a named group
only group "auth"

# Run only tests using specific templates
only req get_user, create_post

# Run only a specific imported file
only import "smoke.bad"

# Skip a test
skip test "legacy" because "deprecated"

# Skip a group
skip group "slow-tests" because "performance"

# Conditional skip inside a test
skip_if status != 200 because "prerequisite failed"

# Fail if condition holds
fail_if status >= 500 because "server error"

Filter Priority

  1. If any only test exists → only those tests run
  2. If no only test but only group → only those groups run
  3. only import limits which imported files execute
  4. only req runs tests that use selected request templates

Config File — .badrc #

A .badrc JSON file can set defaults for all options. CLI flags override config file values, which in turn override in-file defaults.

{
  "base_url":       "https://api.example.com",
  "timeout":        10000,
  "pretty_output":  true,
  "save_history":   true,
  "history_mode":   "all",
  "history_methods": "",
  "history_exclude_methods": "",
  "history_only_failed": false,
  "save_steps":     true,
  "history_include_headers": true,
  "history_include_request_body": true,
  "history_include_response_body": true,
  "history_max_body_bytes": 0,
  "history_dir":    ".bad-history",
  "history_file":   ".bad-history/all-runs.jsonl",
  "history_format": "jsonl",
  "print_request":  false,
  "print_response": false,
  "show_time":      false,
  "show_timestamp": false,
  "json_view":      false,
  "json_pretty":    true,
  "remember_token": false,
  "use_color":      true,
  "fail_fast":      false,
  "log_level":      "info"
}

Use a config file:

./bad file.bad --config .badrc
./bad file.bad --config examples/.badrc
Precedence: CLI flags > in-file settings > .badrc defaults.

History #

With --save, bad writes a JSONL record for each test run. History files enable trend analysis, CI dashboards, and debugging.

Record Fields

FieldDescription
schemaSchema version for the history format
idUnique run identifier
timestampISO 8601 wall clock time
source_fileSource .bad file path
testTest name
test_failedWhether the test failed
requestFull outgoing request snapshot
responseFull response snapshot
stepsStep timeline (if --save-steps)

Advanced Controls

  • history_mode: all, per-file, per-test, off
  • history_methods: include only these methods (CSV)
  • history_exclude_methods: exclude these methods (CSV)
  • history_only_failed: save only failed test records
  • history_include_headers: include/omit request headers
  • history_include_request_body: include/omit request body
  • history_include_response_body: include/omit response body
  • history_max_body_bytes: truncate saved body payload size

Usage

./bad file.bad --save
./bad file.bad --save --save-steps
./bad file.bad --save --save-dir .history
./bad file.bad --save --save-file .bad-history/run.jsonl
./bad file.bad --save --save-mode per-file
./bad file.bad --save --save-mode per-test --save-dir .history
./bad file.bad --save --save-methods GET,POST
./bad file.bad --save --save-only-failed --no-save-response-body
./bad file.bad --save --save-max-body-bytes 2048

Exit Codes #

CodeMeaning
0All assertions passed — success
1One or more assertions failed

Use exit codes in CI pipelines to gate deployments on test results.

./bad tests/smoke.bad && echo "✓ tests passed" || echo "✗ tests failed"

Troubleshooting #

Build Failures

make clean && make

Ensure libcurl development libraries are installed. On Debian/Ubuntu: sudo apt install libcurl4-openssl-dev.

Timeout Errors

Increase the timeout in your file or via CLI:

timeout = 30000             # in file
./bad file.bad --timeout 30000   # via CLI

Import Not Found

Import paths are resolved relative to the current working directory where bad is invoked — not relative to the importing file. Use correct relative paths.

Assertion Path Not Found

Use --print-response or --json-pretty to inspect the actual response shape:

./bad file.bad --print-response --json-pretty

FAQ

  • Is send request the same as send req? Yes. send template is also accepted.
  • Can I use a variable as a request path? Yes — send GET user_path works.
  • How do I merge template body defaults with overrides? Use body_merge true (or merge_body true) inside send req ... with { ... }.
  • Can I skip a test without a block? Yes — skip test "name" because "reason".
  • Can I run against a local server? Yes — set base_url to your local address.
  • Can I pass secrets via environment? Yes — let key = env API_KEY reads from the process environment.
  • How do I run only one test? Use only test "name" at the top of the file.
